# 05 — Database Client

**File:** `src/lib/db.ts`

This tiny file sets up the Prisma database client as a **singleton** — one shared instance across the entire application.

---

## Full File with Line-by-Line Explanation

```typescript
import { PrismaClient } from '@prisma/client'
```
- Imports the auto-generated Prisma client class
- This class is generated by `npx prisma generate` based on `schema.prisma`
- It has typed methods for every model: `db.blogPost.findMany()`, `db.teamMember.create()`, etc.

```typescript
import { PrismaPg } from '@prisma/adapter-pg'
```
- Imports the PostgreSQL driver adapter
- Prisma 7 introduced "driver adapters" — instead of using its own built-in connector, Prisma can use the native `pg` library
- `@prisma/adapter-pg` wraps the `pg` library (the standard Node.js PostgreSQL client)
- Advantage: the `pg` library is battle-tested, well-maintained, and supports connection pooling

---

```typescript
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }
```
- `globalThis` is JavaScript's global object (like `window` in browsers, `global` in Node.js)
- We cast it to include a `prisma` property so TypeScript doesn't complain about us adding a custom property
- `as unknown as { prisma: PrismaClient }` is a "double cast" — TypeScript requires going through `unknown` to change the type completely
- **Why `globalThis`?** See below — it's the singleton pattern.

---

```typescript
function createPrismaClient() {
  const connectionString = process.env.DATABASE_URL!
```
- Reads the database connection string from the environment
- Format: `postgresql://user:password@host:5432/dbname`
- The `!` is a TypeScript "non-null assertion" — tells the compiler "I guarantee this is not undefined"
- If `DATABASE_URL` is actually missing at runtime, this will throw an error when the first DB query is made

```typescript
  const adapter = new PrismaPg({ connectionString })
```
- Creates the `pg` driver adapter with the connection string
- The adapter manages the PostgreSQL connection pool internally
- Connection pooling means multiple requests share a pool of open database connections, which is much faster than opening a new connection for every request

```typescript
  return new PrismaClient({
    adapter,
    log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
  })
}
```
- Creates the Prisma client with the adapter
- `log` controls what Prisma prints to the console:
  - In **development**: `query` (every SQL query), `error`, and `warn` — helps you see what queries are being run
  - In **production**: only `error` — no query logging (would be a security/performance issue)
- `process.env.NODE_ENV` is set to `'production'` by Next.js in production builds automatically

---

```typescript
export const db = globalForPrisma.prisma ?? createPrismaClient()
```
- `??` is the "nullish coalescing" operator — returns the right side if the left side is `null` or `undefined`
- **First request:** `globalForPrisma.prisma` is undefined → calls `createPrismaClient()` → returns new client
- **Subsequent requests:** `globalForPrisma.prisma` is the existing client → returns it directly
- This is the singleton: only one `PrismaClient` ever exists per process

```typescript
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = db
```
- In **development**, saves the client to `globalThis` so it persists across hot-reloads
- **Why only in development?** In production, the Node.js process runs continuously — the module is only loaded once, so `db` is already a module-level singleton. In development, Next.js hot-reloads modules on code changes, which would create a new `PrismaClient` on every reload — eventually exhausting the database connection pool. Saving to `globalThis` (which is NOT hot-reloaded) solves this.
- In production, you don't need this because the process never hot-reloads.

---

## Why Singleton Matters

If you created a new `PrismaClient` on every import:

```typescript
// ❌ BAD — creates a new client (and connection pool) on every import
export const db = new PrismaClient()
```

Every Next.js server function that imports `db` would create its own Prisma instance with its own connection pool. A busy server could easily open hundreds of database connections, overwhelming PostgreSQL.

The singleton pattern ensures the entire app shares ONE Prisma client with ONE connection pool.

## Usage

```typescript
import { db } from '@/lib/db'

// Find all active blog posts, newest first
const posts = await db.blogPost.findMany({
  where: { status: 'published' },
  orderBy: { publishedAt: 'desc' },
})

// Create a new contact message
await db.contactMessage.create({
  data: { name, email, subject, message },
})

// Update a setting
await db.siteSetting.update({
  where: { id: 5 },
  data: { valueKa: 'ახალი ტექსტი', valueEn: 'New text' },
})
```
