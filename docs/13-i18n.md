# 13 — Internationalization (i18n)

**Folder:** `src/i18n/`

The site supports two languages: Georgian (`ka`) and English (`en`). Georgian is the default — it has no URL prefix. English pages get `/en/` in the URL.

---

## File: `src/i18n/routing.ts`

```typescript
import { defineRouting } from 'next-intl/routing'

export const routing = defineRouting({
  locales: ['ka', 'en'],
```
- The two supported locales
- `'ka'` = Georgian (ISO 639-1 language code)
- `'en'` = English

```typescript
  defaultLocale: 'ka',
```
- Georgian is the default — the site's primary audience is Georgian
- The default locale has no URL prefix: `/about` is the Georgian about page

```typescript
  localePrefix: 'as-needed',
})
```
- `'as-needed'` — only add locale prefix when it's NOT the default locale
  - Georgian: `/about`, `/blog`, `/team` (no prefix)
  - English: `/en/about`, `/en/blog`, `/en/team`
- Alternative options: `'always'` (always add prefix), `'never'` (never add prefix)

## File: `src/i18n/request.ts`

```typescript
import { getRequestConfig } from 'next-intl/server'
import { routing } from './routing'

export default getRequestConfig(async ({ requestLocale }) => {
```
- `getRequestConfig` creates a configuration for each individual request
- `requestLocale` — a Promise that resolves to the locale detected from the URL

```typescript
  let locale = await requestLocale
```
- Awaits the detected locale: `'ka'` or `'en'`

```typescript
  if (!locale || !routing.locales.includes(locale as 'ka' | 'en')) {
    locale = routing.defaultLocale
  }
```
- Validates the locale — if it's not in the known list, fall back to the default (`'ka'`)
- This handles edge cases: what if someone requests `/xx/about` (unknown locale)?
- `as 'ka' | 'en'` — TypeScript type cast because `.includes()` expects the specific union type

```typescript
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  }
})
```
- Returns the locale and the translation messages for that locale
- `await import(...)` — dynamic import, loads the JSON file lazily
  - For `'ka'`: loads `src/messages/ka.json`
  - For `'en'`: loads `src/messages/en.json`
- `.default` — JSON files export their content as the default export when imported as modules
- This runs on every server request — next-intl caches it per request

- Creates locale-aware navigation helpers
- `Link` — like Next.js `<Link>` but automatically adds the locale prefix
- `redirect` — like Next.js `redirect()` but locale-aware
- `usePathname` — returns the pathname WITHOUT the locale prefix
- `useRouter` — like Next.js `useRouter` but locale-aware

**Note:** The project mostly uses manual prefix injection (`const prefix = locale === 'en' ? '/en' : ''`) instead of these helpers. Both approaches work; the manual approach is simpler to understand.

## Translation Messages

**Files:** `src/messages/en.json` and `src/messages/ka.json`

These files contain ~40 translation keys for navigation labels and common UI strings.

Example from `en.json`:
```json
{
  "nav": {
    "about": "About",
    "services": "Services",
    "blog": "Blog",
    "team": "Our Team",
    "clients": "Clients",
    "contact": "Contact"
  }
}
```

Usage in a component:
```typescript
const t = useTranslations('nav')
<Link>{t('about')}</Link>  // "About" in English, "ჩვენს შესახებ" in Georgian
```

**Important:** The messages JSON files only contain UI chrome (navigation labels, form labels, button text). All editable page content (hero heading, section titles, contact details) is in the `SiteSetting` database table.

## How the Locale Reaches Components

```
1. Request arrives: GET /en/about
2. Middleware detects locale = 'en' from URL prefix
3. next-intl rewrites: /en/about → internally treated as locale=en, path=/about
4. src/app/[locale]/layout.tsx receives params.locale = 'en'
5. Layout calls getMessages() → loads en.json
6. Layout wraps children in <NextIntlClientProvider messages={messages}>
7. Server components call getLocale() → returns 'en'
8. Client components call useTranslations() → reads from the provider
```

## Language Toggle

When a user switches language, `LanguageToggle.tsx` (a client component):
1. Sets a `NEXT_LOCALE` cookie with `max-age=31536000` (1 year)
2. Does a full-page navigation: `window.location.href = newLocaleUrl`

The full-page navigation (not a client-side route change) is needed so:
- The server renders the page fresh with the new locale
- The middleware picks up the cookie on the next request

## Georgian Language Notes

Georgian (`ka-GE`) uses its own unique script (the Mkhedruli alphabet). The site uses:
- `lang="en"` on the root `<html>` — this is a bug (should be dynamic based on locale)
- Georgian date formatting: `new Date().toLocaleDateString('ka-GE', ...)` → Georgian month names
- The `Geist` font (used in the project) does NOT include Georgian characters — the browser falls back to the system Georgian font. This is acceptable but could be improved by loading a Georgian font like Noto Sans Georgian.
